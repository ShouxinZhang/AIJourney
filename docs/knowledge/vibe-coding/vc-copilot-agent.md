# Agent Mode

## 摘要

Agent Mode 的价值不在“自动写代码”，而在“自动串联步骤”：读代码、改代码、跑检查、补文档，形成可交付闭环。

## 详细内容

### 业务背景

当任务跨越多个文件和步骤时，人工切换上下文成本很高。Agent Mode 通过连续执行把多步任务压缩为一个闭环，适合中等复杂度开发任务。

### 适用场景

- 需要同时改脚本、前端、文档。
- 改动后必须跑构建和测试门禁。
- 需要把过程沉淀到开发日志。

### 管理建议

- 任务先拆成最小可交付（MVP），避免一次性目标过大。
- 明确“可改/不可改”目录，降低误改风险。
- 每轮结束必须有可验证结果（命令输出、文件清单、风险点）。

### 成本与收益

- 收益：交付速度快、过程可追踪、重复动作可复用脚本。
- 成本：需要前置规范（AGENTS、Skill、架构文档）才能稳定发挥。

### 风险与边界

- 无边界约束时，Agent 可能做“超出业务预期”的优化。
- 若忽略验证环节，容易把隐性错误带到后续流程。
